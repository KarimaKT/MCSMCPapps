# ADR 0003 — Entra SSO via Teams Developer Portal SSO registration

| Field | Value |
|---|---|
| Status | accepted (live in main since commits `4b9d497` + `dbd6324`) |
| Date | 2026-05-03 |
| Deciders | engineering agent + user |

## Context

The widget cannot authenticate the user to Copilot Studio Direct Engine inside the M365 Copilot skybridge sandbox: MSAL silent SSO requires reaching `login.microsoftonline.com`, but the sandbox iframe has a null origin and the browser blocks the navigation (`Unsafe attempt to initiate navigation`).

The supported zero-prompt path documented at https://learn.microsoft.com/en-us/microsoft-365-copilot/extensibility/api-plugin-authentication is **Microsoft Entra ID SSO authentication**: the host attaches a user Bearer token to every `/mcp` request, our server validates it, OBO-exchanges for a Power Platform token, and uses that for downstream calls.

The Application ID URI in the token's `aud` claim is **generated by the Teams Developer Portal SSO client registration**, not chosen by the developer. Format: `api://auth-<sso-reg-guid>/<entra-client-id>`. Setting the URI on the Entra app reg to anything else (e.g. `api://<client-id>`) causes `[auth] token rejected: unexpected "aud" claim value` even when everything else is correct.

The Microsoft Enterprise Token Store client (`ab3be6b7-f5df-413d-ac2d-abf1e3fd9c0b`) must be added as a pre-authorized client on the `access_as_user` scope, otherwise the host can't mint tokens for the API at all.

## Decision

**Auth path:** M365 Copilot host → user Bearer (via TDP SSO registration) → our /mcp endpoint → JWT verify against Entra JWKS → OBO exchange (`urn:ietf:params:oauth:grant-type:jwt-bearer`) → Power Platform token (scope `CopilotStudio.Copilots.Invoke`) → server-side CS Direct Engine call.

The Power Platform token does NOT travel to the browser. Per ADR 0001, the widget displays data, not a chat surface.

## Consequences

Easier:
- Zero user-facing auth prompts after one-time tenant admin consent (CDX auto-consents)
- User identity flows end-to-end into CS for licensing and audit
- No MSAL in the widget at all

Harder:
- Setup overhead: Entra app reg + TDP SSO registration + App Service env vars + manifest auth block + admin consent. Documented in `docs/AUTH-ARCHITECTURE.md` (~10 min admin work, one time).
- Each customer tenant repeats this setup. There is no shortcut.
- Client secret rotation is a recurring concern (24-month default expiry). Federated credentials would remove this; not in v0.6 scope.

## Alternatives considered

- **DirectLine 3 token broker** — server mints a DL token; widget connects to `directline.botframework.com`. Eliminates Entra setup but loses user identity (CS sees the bot, not the user). Rejected: violates the "use the user's M365 Copilot license" requirement.
- **Anonymous MCP + browser-side MSAL** — what v0.5 had; fails inside skybridge by design. Rejected (root cause).
- **Federated credentials** — operationally better than client secret; deferred to v0.7 due to setup complexity for first customer.

## Setup checklist (for the customer admin, summary)

See `docs/AUTH-ARCHITECTURE.md` "Entra SSO setup checklist" for full steps. Six steps:

1. Entra app reg: client ID URI + redirect URI + pre-authorized client `ab3be6b7-...` + `CopilotStudio.Copilots.Invoke` permission + admin consent + client secret
2. Teams Developer Portal: register Microsoft Entra SSO client → save SSO registration ID and Application ID URI
3. Update Entra app reg: set `identifierUris` to the URI generated by TDP
4. App Service env vars: `ENTRA_TENANT_ID`, `ENTRA_AUDIENCE`, `ENTRA_CLIENT_ID`, `ENTRA_CLIENT_SECRET`
5. ai-plugin.json: add `auth: { type: OAuthPluginVault, reference_id: <sso reg id> }`; bump manifest; publish
6. Tenant-wide admin consent in M365 admin center → Integrated apps → first publish gets the consent dialog
